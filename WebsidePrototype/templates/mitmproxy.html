https-Traffic mitlesen (am Beispiel von mitmproxy)

Um die übertragenen Informationen zu schützen und sicher zu übertragen nutzen heute die meißten Internetseiten https. Also eine auf SSL/TLS basierende Verschlüsselung.
Das Verfahren vertraut dabei grundsätzlich auf eine Zertifikatsinfrastruktur mit einer Reihe von sog. Root-CAs (Root-Certificate-Authorities), denen die gängigen Standardbrowser vertrauen.
Der Betreiber einer Webseite kann sich von einer solchen Root-CA nun ein Zertifikat für seine URL (bzw. auch für Subdomains) ausstellen lassen. Besucht ein Nutzer nun die Seite, prüft der Browser mittels des Zertifikats deren Authentizität und jeglicher Datenverkehr zwischen Webserver und Browser kann verschlüsselt übertragen. Dritte, die den Datenverkehr mitlesen können (ARP-Spoofing, Provider, etc. ) sehen – wenn browserseitig ausschließlich gesicherte https-Verbindungen akzeptiert werden – somit nur verschlüsselte Daten, die nicht/nur schwer entschlüsselbar sind.


Hat dieser Dritte nun allerdings Zugriff auf das Zertifikat einer beliebigen, im Browser hinterlegten, Root-CA, so kann er sich für beliebige URLs selbst ein Zertifikat ausstellen. Wird im Browser nun eine https-Seite aufgerufen, muss sich der Dritte nur noch dynamisch ein Zertifikat für genau diese Seite erstellen und kann dadurch jeglichen folgenden Datenverkehr mitlesen.


Ein Tool, welches diese Funktionalität beinhaltet ist mitmproxy [https://github.com/mitmproxy/mitmproxy] (es gibt noch viele weitere Tools, mit teils unterschiedlichen Funktionen, wie z.b. Burp Suite, Charles, sslsplit, usw.). Neben diesem Feature bietet mitmproxy z. B. auch die Möglichkeit Pakete auf ihrem Weg zu manipulieren.


Obwohl die Verschlüsselung mittels SSL/TLS grundsätzlich sicher ist, sind dennoch einige Beispielszenarien denkbar in denen dieses Verfahren möglich wäre:
- Die Zertifkatsstelle/Zertifikatsvergabe wurde kompromittiert

Ein unautorisierter Dritter kann sich a) ein Zertifikat für eine Website ausstellen lassen oder b) sich selbst beliebige Zertifikate ausstellen.
- Besitz des Serverzertifikat wurde erlangt

Ein Dritter hat Zugang zum privaten Schlüssel des Webservers erlangt.
- Zugriff auf den Opfer-PC → lokal Vertrauen gegenüber neuen Zertifikaten aussprechen

Ein Dritter hat die Möglichkeit einer neuen Root-CA das Vertrauen auszusprechen.
- praktisch: Analyse von SSL/TLS-Verschlüsseltem Traffic von unbekannten Anwendungen (so lange kein Pinning verwerden wird…)

Ziel hierbei ist es zu analysieren, welche Daten von dieser Anwendung (z. B. auch Android/IOS-Apps, da diese oftmals über https mit ihren Servern kommunizieren) übertragen werden.


Mimtproxy agiert als transparenter Proxy zwischen Webserver und Browser. Effektiv werden also zwei getrennte https-Verbindungen erstellt, was die Grafik von mitmproxy.org zeigt.


In Worten: Der TCP-Verbindungsaufbau des Client / Browsers wird (via iptables) an mitmproxy weitergereicht. (1,2)

Auf den vom Client / Browser initiierten SSL-Handshake (3) wird seinerseits von mitmproxy ein SSL-Handshake zum eigentlichen Webserver erstellt (4,5). Anschließend wird der SSL-Handshake zwischen mitmproxy und Client / Browser mit gefälschtem Zertifikat abgeschlossen (6) und Requests laufen transparent zwischen Browser / Client und Webserver über mitmproxy. (7,8)



Demo 1 – Mitmproxy einrichten

Schritt 1 - (Optional) mitmproxy installieren

sudo apt-get install python-pip python-dev libffi-dev libssl-dev libxml2-dev libxslt1-dev libjpeg8-dev zlib1g-dev

sudo pip install mitmproxy

wichtig: mitmproxy mittels pip, nicht mit apt-get installieren, da hier eine veraltete Version hinterlegt ist.


Schritt 2 - mitmproxy starten

sudo mitmproxy


Schritt 3 – Proxy in Firefox einrichten

Auf dem Ziel-PC in Firefox die <IP-Adresse des mitmproxy-PC> mit Port 8080 als Proxy eintragen:

Bearbeiten - Einstellungen – Erweitert – Netzwerk – Einstellungen – manuelle Proxy-Konfiguration

<IP-Adresse des mitmproxy-PC> mit Port 8080 eintragen und "für alle Protokolle diesen Proxy-Server verwenden" auswählen und mit "ok" speichern.


-------------------------------------

Alternativ über transparanten Proxy – v.a. zum testen von Anwendungen besser:

Schritt 2 - mitmproxy starten

sudo mitmproxy -T –host


Schritt 3 - Vorbereitungen, damit die Pakete an mitmproxy weitergeleitet werden

sudo iptables -t nat -A PREROUTING -i eth0 -p tcp --dport 80 -j REDIRECT --to-port 8080

sudo iptables -t nat -A PREROUTING -i eth0 -p tcp --dport 443 -j REDIRECT --to-port 8080


Schritt 4 - Netzwerktraffic über den mitmproxy-PC leiten

(Eingabe auf dem Client; Einstellungen sind temporär und werden beim Neustart entfernt; Der Einfachheit halber machen wir dies durch Änderung des Default-Gateways auf dem Ziel-PC; aufwändigere Alternative: ARP-Spoofing)


sudo route del default

sudo route add default gw <IP-Adresse des mitmproxy-PC>

-------------------------------------


Schritt 5 - mitmproxy Zertifikat auf den Ziel-PC kopieren (z. B. via scp)

Nach der Installation von mitmproxy findest sich das Zertifikat im Pfad (v. 0.15): ~/.mitmproxy/mitmproxy-ca-cert.cer


Schritt 6 - Zertifikat im Browser einbinden

Firefox - Bearbeiten - Einstellungen - Erweitert - Zertifikate - Zertifikate anzeigen - Zertifizierungsstellen - Importieren.. - mitmproxy-ca-cert.cer auswählen – öffnen


Hinweis: Dieser Schritt sollte nur auf Labor-VMs durchgeführt werden, da im dadurch jeglicher https-Datenverkehr uneingeschränkt von einem Dritten mitgelesen werden kann!
